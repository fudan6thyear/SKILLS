# 公司标准 Data Flow 完整参考手册

本文件是公司标准 Data Flow 的唯一参考文档，整合了格式规范、生成规则、校验清单和迭代经验。

---

# Part 1: 格式规范

## 1.1 目的与适用范围

本规范定义公司标准 `Data Flow` 图的统一格式，供 `draw.io`/`diagrams.net` 及 AI 生成流程使用。适用于业务活动个人信息流转梳理、系统间数据采集/使用/传输/归档分析、隐私评估/合规评审/架构说明等场景。

## 1.2 输出目标

每张标准图输出为一个 `.drawio` 文件，满足：统一的二维矩阵布局、明确的数据生命周期阶段、明确的参与方/内部人员/内部系统/第三方边界、统一图例和编号方式、箭头连线表达数据流向。

## 1.3 页面固定结构

- **标题区**：页面顶部左上角，格式固定为 `业务名称——活动名称`，单行不换行。
- **主图区**：二维矩阵，横向表示数据生命周期阶段，纵向表示不同参与主体和系统边界。
- **右侧图例区**：固定保留，包含图形类型说明、数据项编号清单、连线颜色说明。

## 1.4 横向阶段规范

横向固定为 4 个一级阶段：`信息收集` / `存储/使用` / `分享/传输` / `归档/删除`。

`存储/使用` 阶段允许细分子列（如 `用户界面`、`部门`、`业务系统`），子列仅出现在该阶段，标题置于一级阶段下方，数量控制在 2~4 个。

每个一级阶段使用浅灰色标题条。即使某阶段无内容也保留空列。画布行高、列宽随实体数量与尺寸自适应（生成器 v7），不使用固定像素画布。

## 1.5 纵向泳道规范

纵向固定为 4 类泳道：`数据主体` / `内部人员` / `内部系统` / `第三方`。

放置规则：

| 泳道 | 放置内容 |
|------|---------|
| 数据主体 | 用户、注册用户、客户、员工等自然人主体 |
| 内部人员 | 运营、客服、管理员、审核员、部门角色等人工参与方 |
| 内部系统 | 页面、前端入口、业务系统、后台系统、数据库、对象存储等 |
| 第三方 | 外部服务、合作方系统、第三方云服务、外部接口等 |

所有业务元素必须放置在四类泳道之一中，禁止混放。

## 1.6 图形与符号规范

| 元素类型 | 推荐图形 | draw.io 样式 |
|---------|---------|-------------|
| 人工操作/第三方（非云服务） | 圆角矩形 | `rounded=1` |
| 界面/功能名称 | 普通矩形 | 默认矩形 |
| 后台/系统/数据库集 | 圆角矩形 | `rounded=1` |
| 文档/数据库 | 圆柱形 | `shape=cylinder3` |
| 第三方云服务 | 云朵形 | `shape=cloud` |
| 数据项编号 | 彩色编号圆点 | `ellipse` |
| 数据主体（用户） | 小人图形 | `shape=umlActor` |

同类对象必须使用同一图形，图形边框/字体/字号保持一致。

## 1.7 数据项编号规范

- 每种数据项对应唯一编号（阿拉伯数字 1~N）
- 同一编号可在多条路径中重复出现
- 右侧必须列出所有编号的数据项说明（编号 + 数据名称 + 简要说明）
- 图中出现的每个编号必须在右侧清单中有定义，反之亦然

## 1.8 连线与流向规范

- 所有数据流使用有方向的箭头
- 连线默认采用直角折线
- 连线应避免穿透实体图形
- 多种数据项在同一路径上传输时复用一条主线，通过沿线彩色编号圆点表达
- 一张图只表达一个数据处理活动时，所有主流转线使用同一 `activity_color`
- 数据项差异通过编号圆点颜色区分，不通过连线颜色区分

## 1.9 生命周期覆盖要求

每张图应覆盖完整数据生命周期（信息收集 → 存储/使用 → 分享/传输 → 归档/删除）。若某阶段不存在，保留空列。

## 1.10 AI 生成必填输入

`business_name`、`activity_name`、`activity_color`、数据主体、涉及页面/功能、内部人员角色、内部系统/数据库、第三方服务、数据项清单、每条数据流的来源/去向/所属阶段、是否存在分享/传输、是否存在归档/删除。

# Part 2: 生成规则

## 2.1 度量基准

所有间距规则使用 `D = DOT_SIZE = 24px`（一个数据圆点的直径）作为基准单位。

## 2.2 布局规则

### 行列投射 (HC-1)

每个实体必须放在其声明的泳道（Y 范围）和阶段（X 范围）内。生成器的 `place_entities()` 使用 `lane_b[lk]` 和 `stage_b[stage]` 确定范围。

### 连线间距公式 (HC-2)

```
gap_required = (N + 2) * D
```

N = 该 flow 的数据圆点数。若折线且圆点分布在多个方向：`gap_per_direction = (n_i + 2) * D`。

当 `gap_actual < gap_required` 时，依次：增大 `SLOT_H` → 增大 `stage_w` → 使用 `layout_hint.x/y`。

### 布局优先于连线 (HC-3)

修复优先级严格为：1. 调实体位置 → 2. 调线路 → 3. 调圆点。禁止"只调线不调位"。

### 平行连线间距 (HC-4)

```
parallel_gap >= 2 * D = 48px
```

### 四面端口分配 (HC-5)

实体 <=4 条连线时：每面最多 1 条。>4 条时：共用面的线距实体至少 `2D`。

生成器使用形状感知端口表：

| 形状 | Right | Left | Top | Bottom |
|------|-------|------|-----|--------|
| umlActor | (1.0, 0.33) | (0.0, 0.33) | (0.5, 0.0) | (0.5, 1.0) |
| cloud | (0.875, 0.5) | (0.08, 0.53) | (0.5, 0.15) | (0.5, 0.85) |
| cylinder | (1.0, 0.5) | (0.0, 0.5) | (0.5, 0.0) | (0.5, 1.0) |
| rect | (1.0, 0.5) | (0.0, 0.5) | (0.5, 0.0) | (0.5, 1.0) |

## 2.3 自适应画布 (v7)

- **行高**: `max(MIN_LANE_H, SLOT_PAD_TOP + max_entities_in_column * SLOT_H + SLOT_PAD_BOT)`
- **列宽**: 非`存储/使用` → `max(MIN_STAGE_W, widest_entity + margin)`；`存储/使用` → 子列宽度累加
- 常量: `SLOT_H=104`, `SLOT_PAD_TOP=20`, `SLOT_PAD_BOT=50`
- 添加实体后应重新生成，不手动改回固定尺寸

## 2.4 连线路由规则 (v6+)

### 方向选择

- `_choose_direction()` 枚举所有有效方向对
- L 弯优先：垂直方向对（如 `bottom->right`）获得 -50 评分奖励，仅需 1 个航点
- 平行方向对（如 `right->left`）产生 S/Z 弯，2 个航点
- 语义偏好 (v8.1/v8.2)：
  - 回流到上游 UI：同泳道对角线优先 `left->bottom`
  - 跨阶段回到早期 UI：优先 `top->top`
  - 多圆点写入 datastore：优先 `bottom->left/right`

### 拐点约束 (v6)

- 0 拐点（直线）：对齐且 <= 3 个圆点
- 1 拐点（L 弯）：垂直方向对
- 2 拐点（S/Z 弯）：平行方向对
- 2 拐点（弧）：`n_dots > 3` 时弧深 = `n_dots * 14px`
- 常规边最多 2 拐点

### 碰撞规避 (v8)

- `compute_route()` 接收 `placed`（实体障碍）和 `used_paths`（已有路径）
- 评分维度：实体碰撞数、路径重叠数、拐点数、圆点空间不足量、重复边惩罚、路径长度
- `_path_respects_ports()` 过滤逆向路径
- 重复 `(source, target)` 边递增端口偏移

## 2.5 圆点放置规则 (v8)

### 线段选择

`_plan_dots_for_path()` 评分：

- 内段（非首尾）：`-120` 奖励
- 外段（首/尾）：`count > 1` 时 `+1200` 惩罚
- 紧凑惩罚：`(needed - actual) * 120`
- 实体碰撞：`+50000/hit`，圆点重叠：`+10000/hit`，线段碰撞：`+400/hit`

### 短路径缩放

单圆点路径 < 90px 时，圆点缩小到 `max(18, min(24, seg_len/3))`。

### 聚簇

同一 flow 的圆点聚簇在一个线段上，间距 `DOT_CLUSTER_GAP = 20px`，不沿整条路径铺开。

# Part 3: 验证

## 3.1 验证闭环工作流

生成后必须通过自动化验证才能交付：

1. 主 agent 生成 `.drawio`
2. 主 agent 通过 Cursor `Task` 工具启动 `shell` subagent 运行验证脚本：
   ```
   python "scripts/validate_company_data_flow.py" "<drawio_path>" "<json_path>"
   ```
3. Subagent 返回 JSON 报告
4. 全部 PASS → 交付；任一 FAIL → 修复后重新生成，禁止带已知缺陷交付

### 验证检查项

| ID | 名称 | 检查内容 |
|----|------|---------|
| HC-1 | 行列投射 | 实体 (x,y) 在泳道 Y 范围和阶段 X 范围内 |
| HC-2 | 连线间距(N+2)D | 源-目标距离 >= (N+2)*24px |
| HC-3 | 布局优先于连线 | SKIP（流程约束） |
| HC-4 | 连线间距>=2D | 平行线段间距 >= 48px |
| HC-5 | 四面端口优先 | 总连线 <=4 时每面最多 1 条 |
| CK-1 | 页面结构 | 4 阶段、4 泳道、标题、图例存在 |
| CK-2 | 图形类型 | shape style 匹配实体 type |
| CK-3 | 数据圆点 | 圆点标签匹配 data_items |
| CK-4 | 连线完整性 | 每条 flow 都有对应 edge |
| CK-5 | 拐点约束 | 每条边航点数 <= 2 |

### 失败修复优先级

1. HC-1 → 修正 JSON 中的 `stage`/`lane_key` 或 `layout_hint`
2. HC-2 → 增大 `SLOT_H`/`stage_w` 或添加 `layout_hint` 拉大间距
3. HC-4 → 调整实体位置制造分离空间
4. HC-5 → 重新分配连线到不同端面
5. CK-* → 修正 JSON 输入

## 3.2 校验清单（人工复核用）

### 页面结构

- [ ] 标题存在，格式为 `业务名称——活动名称`
- [ ] 4 个一级阶段存在
- [ ] 4 个固定泳道存在
- [ ] 右侧独立图例区存在
- [ ] 缺失阶段保留空列
- [ ] 自适应版式：空泳道不过大、密集泳道/阶段有足够空间
- [ ] 图例区在主图右侧且不重叠

### 图形类型

- [ ] 数据主体使用小人图形
- [ ] 内部人工角色使用人工或圆角矩形
- [ ] 页面/功能使用矩形
- [ ] 内部系统使用圆角矩形
- [ ] 数据库使用圆柱形
- [ ] 第三方云使用云朵形
- [ ] 数据项使用编号圆点
- [ ] 同类对象使用同一图形

### 实体放置

- [ ] 各类实体只出现在对应泳道
- [ ] 同一对象尽量只出现一次
- [ ] 无节点重叠或过小间距

### 数据项编号

- [ ] 所有数据项唯一编号
- [ ] 同编号只对应一种数据项
- [ ] 图中编号与右侧清单双向一致

### 连线

- [ ] 所有连线有明确方向
- [ ] 优先使用直角折线
- [ ] 连线不穿过图形本体
- [ ] 连线真实贴附图形边缘
- [ ] 特殊图形接点自然可读
- [ ] 返回流/跨泳道流不切穿主阅读区
- [ ] 常规边 <= 2 拐点

### 数据圆点

- [ ] 圆点落在线上
- [ ] 圆点在流转线之上（不被线盖住）
- [ ] 多圆点优先聚簇
- [ ] 圆点不遮挡箭头、名称或拐点

### 颜色与图例

- [ ] 单活动单颜色
- [ ] 图例说明颜色含义
- [ ] 圆点颜色不影响编号识别

### 生命周期

- [ ] 覆盖信息收集、存储/使用
- [ ] 如适用，覆盖分享/传输、归档/删除
- [ ] 不适用的环节保留空列

### 结论分级

- `通过`：全部关键项满足
- `需修改`：结构正确但存在图例/编号/命名/布局问题
- `不通过`：缺失固定结构/泳道错误/无编号清单/生命周期不完整

# Part 4: 迭代经验 (v1–v9)

## 4.1 从用户手工修改 v4 识别的设计意图

### 高置信意图

- 连线必须"看起来真的接上了图形"
- 对 `umlActor` 和 `cloud`，接点应更自然
- 折线应走可解释的通道
- 数据圆点表达"数据束"，不是路径采样点
- 圆点贴着关键线段出现，不平均铺满
- 拐点 <= 2 个，圆点过多时适当增加
- 优先选择最近的连接点

### 中置信意图

- 主处理区域可适度强调
- 局部节点尺寸可为可读性做小幅让步

## 4.2 v1–v6 演进

| 版本 | 核心修正 |
|------|---------|
| v1 | 初始版本（多颜色、交叉多、圆点不在线上、无用户图形） |
| v2 | 统一颜色到流向类别、直角折线、圆点沿路径、`umlActor` |
| v3 | 同活动=同颜色、编号顺序配色、gutter/通道规则、固定槽高 |
| v4 | 显式 `entry/exit` 锚点、圆点后绘制置顶、中段聚簇 |
| v5 | 形状感知端口系统、3~4 航点跨泳道路由、重复边渐进偏移、`exitPerimeter=0` |
| v6 | 垂直方向对（L 弯）优先、严格 <=2 拐点、移除 stub |

## 4.3 划线问题根因分析

### v5 识别的 7 个根因

1. 只有 4 个刚性锚点 → 形状感知端口表
2. 所有折线只有 2 个航点 → 增加到 3~4 个（v5），后精简到 <=2（v6）
3. 相同 source-target 对叠压 → 递增端口偏移
4. `exitPerimeter=1` 硬编码 → 特殊图形设 `perimeter=0`
5. 缺少引导段 → v5 增加后 v6 移除
6. 返回流只有 2 点底部通道 → v5 用 4 点 U 形后 v6 改为 L 弯/S 弯
7. 锚点计算不一致 → 统一 `exit_dir`/`entry_dir` 传递

### v6 识别的 2 个核心缺陷

A. `_choose_direction()` 只产生平行方向对 → 增加垂直方向对候选
B. `compute_route()` 航点过多 → 严格限制 <=2，移除 stub

### v7 识别的 1 个核心缺陷

C. 画布行高列宽固定 → `build_layout()` 按 `(lane, stage, subcolumn)` 动态计算

### v8 识别的 2 个核心缺陷

D. 候选路径无障碍物检测 → `_path_collision_count()`、`_path_overlap_count()` 综合评分
E. 圆点落位无跨边避让 → `_plan_dots_for_path()` 关键线段优先 + 碰撞评分

### v8.1 识别的语义偏好

F. 回流到上游 UI 只按几何最短选面 → 增加 `left->bottom` 偏好

### v8.2 识别的特殊场景

G. 跨阶段回流/竖向写 datastore → `top->top` 偏好 + 侧向两拐点候选

## 4.4 v9 硬约束整合 + 验证闭环

### 来源

用户核心诉求："为什么你就是不会调整各个图形的相对位置，而是一味地调整线条？" —— 暴露了 v1-v8.2 的结构性盲区。

### 5 条硬约束

| ID | 约束 | 与现有规则的关系 |
|----|------|-----------------|
| HC-1 | 行列投射 | 新增。之前靠自动放置隐式保证但从未验证 |
| HC-2 | 连线间距 = (N+2)D | 新增。之前圆点落位只做后置检查 |
| HC-3 | 布局优先于连线 | 新增。对 v1-v8.2 的根本性纠正 |
| HC-4 | 平行连线间距 >= 2D | 量化升级。v8 只做 boolean 重叠检测 |
| HC-5 | 四面端口优先 | 量化升级。v5 端口表只做方向选择 |

### 验证闭环引入原因

AI 缺乏视觉能力，无法"看图"发现违规。新建 `validate_company_data_flow.py` 程序化检查。

### 对生成器的后续要求

- `place_entities()` 应按 flows 圆点数预检间距
- `build_layout()` 应按 flow 反推最小间距
- `compute_route()` 应量化平行线间距（非 boolean）
- 端口分配应预统计各面连线数

# Part 5: 工作流与反模式

## 5.1 默认执行路径

1. 读取或构建结构化输入
2. 若无输入文件，从模板创建
3. 运行生成器生成 `.drawio`
4. 启动验证 subagent 检查报告
5. 验证失败则修复并重新生成（回到步骤 3）
6. 交付经验证的 `.drawio`
7. 若用户手工编辑，比较差异、识别意图、更新规则

## 5.2 用户手工编辑分类

- **语义修正**：实体/阶段/数据项/图例语义错误
- **视觉修正**：连线贴附、路由通道、圆点放置、间距
- **偏好信号**：强调、路径风格、紧凑度、可读性

只在意图可跨图复用时才回写项目规则。

## 5.3 反模式（Anti-Patterns）

- 禁止只调线不调位（空间不足时必须先移实体）
- 禁止圆点压在实体 bounding box 上
- 禁止平行线间距 < 2D (48px)
- **禁止跳过 subagent 验证直接交付**
- 禁止同一活动使用多种连线颜色
- 禁止用 Mermaid 代替原生 draw.io XML
- 禁止把圆点当作路径均匀装饰
- 禁止忽略用户手工编辑（应检查是否可复用）
- 禁止改回固定画布尺寸

## 5.4 项目推荐表达

- `activity_name`：数据处理活动名称
- `activity_color`：该活动在整张图中的唯一连线颜色
- `data_items`：数据项字典
- `flows`：实体之间的业务流向

不要再把 `line_category` 或"每条边独立颜色"作为主设计中心。
